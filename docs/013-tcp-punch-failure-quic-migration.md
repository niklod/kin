# Decision 013: Отказ от TCP hole punching, переход на QUIC

**Date:** 2026-04-05

## Проблема: TCP hole punching ненадёжен

### Что такое TCP simultaneous open

Реализованный в Phase R механизм NAT traversal основан на TCP simultaneous open: оба пира одновременно делают `connect()` друг к другу с одного и того же порта. При удаче TCP-стек на обоих концах воспринимает входящий SYN как ответ на свой исходящий SYN, и соединение устанавливается без явного `listen()`.

### Почему это не сработало

Тест с реальными пользователями за домашними роутерами показал стабильный сбой:

```
dial tcp :7777->159.89.26.169:7777: connectex: No connection could be made
because the target machine actively refused it.
```

**Первопричина:** Для cone NAT нужно, чтобы слушающая сторона (listener) создала запись в своей NAT-таблице раньше, чем SYN от инициатора дойдёт до её роутера. Эта запись создаётся только тогда, когда сторона сама отправляет пакет с нужного порта в сторону пира.

Проблема в том, что relay отправляет rendezvous обоим пирам одновременно, но время доставки разное (зависит от задержки до relay). В нашем тесте:

- Инициатор получил rendezvous: `00:55:31.482`
- Слушатель получил rendezvous: `00:55:31.564` — на **82мс позже**

Инициатор выслал SYN немедленно. Слушатель ещё не успел «пробить» свой NAT. Роутер слушателя получил SYN от неизвестного адреса → RST.

### Попытка исправить: priming + задержка

Мы добавили `nat.PrimeNAT` (слушатель отправляет SYN к инициатору при получении rendezvous) и `punchPrimeDelay = 300ms` (инициатор ждёт перед отправкой SYN). Это хрупкое решение:

- Успех зависит от разницы задержек до relay у обоих пиров
- 300мс — произвольная константа, нет гарантий
- При плохом интернете или большой разнице в пинге всё равно упадёт

### Фундаментальные ограничения TCP punch

1. **Symmetric NAT** — не поддерживается в принципе. Symmetric NAT выдаёт разный внешний порт для каждого нового destination. TCP simultaneous open требует знать точный внешний порт заранее.

2. **Timing race** — между получением rendezvous и отправкой SYN есть окно, которое нельзя надёжно синхронизировать без общего сервера времени.

3. **Stateful NAT** — многие роутеры используют stateful inspection и отклоняют входящие SYN, даже если исходящий SYN уже был отправлен, если соединение не прошло полный TCP handshake в правильном порядке.

4. **TCP RST** — роутеры могут активно отвечать RST на неизвестные входящие соединения, что мгновенно завершает попытку вместо ожидания таймаута.

## Решение: QUIC (UDP)

### Почему UDP hole punching надёжнее

UDP не имеет состояния соединения. Для создания NAT-записи достаточно отправить один UDP-пакет в сторону пира — запись создаётся немедленно, без необходимости получить ответ. Нет требования simultaneous open, нет timing race.

Алгоритм UDP hole punching:

1. Оба пира получают external addr друг друга от relay (как сейчас)
2. Оба одновременно отправляют UDP-пакеты навстречу → в NAT-таблицах обеих сторон появляются записи
3. Инициатор делает QUIC `Dial` → QUIC Initial packet проходит через уже открытую «дыру»
4. Слушатель принимает QUIC connection

Успех не зависит от того, кто первый: достаточно чтобы оба отправили UDP-пакеты примерно одновременно. Нет RST (UDP stateless), нет timing race.

### Почему QUIC, а не голый UDP

QUIC даёт гарантированную доставку, мультиплексирование потоков и встроенный TLS 1.3 поверх UDP. Это позволяет:

- Переиспользовать существующую TLS-идентификацию (Ed25519 сертификаты)
- Не реализовывать надёжную доставку поверх UDP самостоятельно
- Использовать `quic-go` — зрелую Go-библиотеку

### Что не меняется

- Relay-сервер: только signaling, протокол не меняется, транспорт TCP остаётся
- Protobuf-протокол между пирами
- TLS-идентификация (Ed25519 keypair, self-signed cert)
- Invite-система
- PeerStore

### Что меняется

| Компонент | Сейчас | После |
|---|---|---|
| `internal/transport/` | TCP + TLS | QUIC (`quic-go`) |
| `internal/nat/punch.go` | TCP simultaneous open | UDP priming + QUIC dial |
| `internal/nat/sockopt_*.go` | SO_REUSEPORT | Удаляются |
| `internal/transport/listenconfig_*.go` | SO_REUSEPORT для listener | Удаляются |
| `internal/connmgr/` | punchPrimeDelay, PrimeNAT | Упрощается |
